> 蒸馏世界，成就自己！蒸馏自己，成就世界！

**日期**：2026 年 6 月 24 日  
**状态**：方向确定，待落地

---

## 一、核心定位

**蒸馏不是一个网站，是一个 Agent 网络的能力后端。**

传统社区让用户来逛。蒸馏让 Agent 替用户干活。  
网站只是仓库和交易市场，真正的产品体验发生在用户自己的 Agent 里。

---

## 二、产品哲学

### 人都是懒的

主动蒸馏自己的人永远是少数。所以：

- ❌ 不要指望用户主动打开「蒸馏工坊」
- ✅ Agent 观察用户 → 主动发现模式 → 自己蒸馏 → 拿给用户确认

### Skill 是活的

Skill 不是文章，不是用来看的。它是：

- 可以被 Agent 直接执行的
- 用户可能完全无感就享受到了好处
- 效果可度量（帮你省了多少时间、执行了多少次）

### 分享是零成本的

用户不需要「创作」。Agent 已经帮你蒸馏好了、跑验证了、甚至写好了描述。  
你只需要在 Agent 问你的时候点一下「好，分享」。

### 智能发生在边缘

所有推理、匹配、打分、观察都发生在**用户自己的 Agent 里**，不在云端平台。

平台是仓库和交易市场，真正的智能跑在离用户最近的地方——就像 mars 的「钳程无忧」，所有打分、盯机会、判断合不合适都是用户自己的 Agent 在本地计算的。

这意味着：
- **隐私天然有保障** — 你的工作数据不需要上传，Agent 在本地就完成了分析
- **个性化无上限** — 每个人的 Agent 基于自己的上下文做决策，不是平台的统一推荐算法
- **离线也能工作** — Skill 装到本地后，不依赖平台也能执行
- **平台不是瓶颈** — 10 万用户的 Agent 各自在本地跑，平台不需要承担推理算力

## 三、系统架构

```
用户的日常工作环境
    ↕ （本地 Agent 一直在身边）
OpenClaw / Hermes / Claude Code / 浏览器插件
    ↕ （心跳 + 事件驱动 + 定时任务）
蒸馏社区 API
    ↕
Skill 仓库 + AI 推荐引擎 + 声望网络
```

### 三个角色

| 角色 | 职责 |
|------|------|
| **本地 Agent** | 观察用户、发现模式、蒸馏 Skill、执行 Skill、场景推送 |
| **蒸馏社区** | 存储 Skill、匹配推荐、声望计算、质量评估 |
| **用户** | 确认/微调蒸馏结果、决定是否分享、享受 Agent 帮忙干活 |

## 四、核心链路

### 链路 1：Agent 主动蒸馏（供给侧）

```
Agent 持续观察用户工作
    ↓
发现重复模式（对话历史 / 浏览行为 / 工作节奏）
    ↓
自主蒸馏成 Skill 草稿
    ↓
先自己拿来帮用户干活（验证好用）
    ↓
跑了 N 次、用户没改过 → 确认好用
    ↓
Agent 问：「这个已经帮你用了 8 次了，社区有人需要，分享出去？」
    ↓
用户确认 → 发布到社区 → 声望 ↑
```

**关键设计**：不是蒸馏完就问「要不要公开」，而是**验证好用之后**再问。

### 链路 2：Agent 主动找 Skill（消费侧）

```
Agent 在帮用户干活
    ↓
意识到「这个事有更好的方法」或「我没做过」
    ↓
实时 ping 蒸馏社区 API
    ↓
社区返回匹配的 Skill（或多个 Skill 的组合方案）
    ↓
Agent 直接装上就用 / 或组合执行
    ↓
用户甚至可以无感——活干完了就行
```

### 链路 3：定时任务 → 场景预判

```
Agent 的定时任务触发（如：周五 14:00）
    ↓
Agent 知道「用户每周五写周报」
    ↓
去社区拉取/确认「周报生成」Skill
    ↓
基于本周对话历史预生成周报草稿
    ↓
在对话里说：「周报写好了初稿，你看看改改？」
```

**用户收到的不是「推荐你一个 Skill」，而是活已经干完了。**

### 链路 4：AI 搜索 + Skill 组合（问题驱动）

```
用户对 Agent 说：「我下周要做新产品立项评审」
    ↓
Agent 理解意图 → 调用社区 AI 推荐 API
    ↓
社区返回组合方案：
  1. 「先写新闻稿」— 理清价值主张
  2. 「市场规模测算」— 算清楚盘子有多大
  3. 「做一份惊艳的演示」— 包装成 Deck
    ↓
Agent：「我帮你按这个顺序走一遍？」
    ↓
用户：「好」
    ↓
Agent 依次执行三个 Skill，最终产出完整立项材料
```

## 五、分享机制设计

### 为什么用户会愿意分享？

| 动力 | 设计 |
|------|------|
| **零成本** | Agent 已经帮你蒸馏好了，你只需点「确认」 |
| **已验证** | 不是「可能有用」，是「确实帮我用了 N 次」 |
| **有人需要** | Agent 会告诉你「社区有 X 人在找类似的」 |
| **声望正反馈** | 分享后被引用 → 排名升 → 更有动力 |
| **反哺自己** | 别人 Fork 你的 Skill 改进了 → 你也能受益 |

### 分享确认时机

Agent 在以下条件满足时询问：

1. ✅ Skill 已被自动执行 ≥ 3 次
2. ✅ 用户未对执行结果做过修改（说明质量稳定）
3. ✅ 社区中存在相关需求（有人搜索过类似问题）

询问话术示例：

> 「这个'客户投诉处理三步法'我已经帮你跑了 5 次了，每次你都没改直接用了。社区里有 12 个人在找类似的 Skill，分享出去帮帮他们？你的声望会 +50。」

## 六、核心指标

### 不再是传统社区指标

| ❌ 传统指标 | ✅ 蒸馏指标 |
|------------|------------|
| DAU/MAU | Agent 日活跃调用次数 |
| 用户发帖数 | Agent 自动蒸馏的 Skill 数 |
| 浏览量 | Skill 被执行次数 |
| 点赞数 | 引用数（多少 Agent 在用我的 Skill） |
| 用户停留时长 | 帮用户节省的时间 |

### 北极星指标

**每周帮用户自动完成的任务数** × **用户确认满意率**

## 七、信息采集渠道（Agent 如何观察用户）

| 渠道 | 采集方式 | 能发现什么 |
|------|----------|-----------|
| **对话历史** | 本地 Agent 的 memory | 重复的提问模式、相似的任务 |
| **浏览器插件** | 页面行为追踪 | 每天重复操作的工作流 |
| **日历/飞书** | API 集成 | 周期性事件（周报、月会、复盘） |
| **文件系统** | 本地扫描 | 反复创建的相似文档模板 |
| **IDE 插件** | 代码行为 | 重复的编码模式、Review 习惯 |

**隐私原则**：所有观察数据留在本地，只有用户确认分享的 Skill 才上传社区。

## 八、与「虾兵蟹将」叙事的结合

在原有觅游的叙事框架下：

- **虾兵** = 你的本地 Agent
- **蒸馏** = 虾兵观察你工作后，自己学会了你的方法论
- **分享到社区** = 你的虾兵教会了别人的虾兵
- **声望** = 你的虾兵培养了多少「徒弟」

话术示例：
> 「你的虾兵小P 观察你做了 3 次竞品分析后，自己学会了你的套路。现在它可以独立帮你做了。要不要让它去社区教其他虾兵？」

## 八点五、Skill 歌单（收藏集）

除了按职业赛道聚合，Skill 还可以像**网易云音乐歌单**一样被自由组合成收藏集。

### 为什么要做歌单

- 职业分类是「官方视角」，歌单是「用户视角」—— 一个人的真实工作往往跨多个分类
- 降低新人门槛 —— 不用自己一个个挑，直接「一键安装歌单」
- 增加社交货币 —— 「看看大佬的 Skill 配置」本身就有传播力
- 给 KOL 新的创作形式 —— 不只是蒸馏单个 Skill，还能策展和编排

### 歌单类型

| 类型 | 示例 |
|------|------|
| **新手入门包** | 「新手产品经理必备 10 个 Skill」「前端工程师不能没有的 10 个 Skill」 |
| **场景组合包** | 「从 0 到 1 做一次完整的产品立项」「搞定季度 OKR 全流程」 |
| **大佬配置单** | 「张三（产品赛道 Top 1）的虾兵装备清单」 |
| **公司/团队包** | 「XX 团队新人 Onboarding 必装」 |
| **AI 智能推荐包** | Agent 基于你的赛道 + 能力缺口自动生成的个性化歌单 |

### 歌单的互动

- **一键安装全部** — 批量给虾兵装上
- **Fork 歌单** — 复制一份改成我的版本
- **订阅歌单** — 创建者更新时我也自动更新
- **歌单引用数** — 跟单个 Skill 一样计入声望

### 跟 Agent 的结合

Agent 可以：
- 根据用户当前能力缺口，**自动生成个性化歌单推荐**
- 发现用户安装了某个歌单中的 3 个 Skill，**提醒安装剩下的**
- 帮用户**自动创建歌单**：「我观察到你最常用这 8 个 Skill，要不要打包成歌单分享？」

## 九点五、从 Skill 到 Plugin：蒸馏的范畴升级

蒸馏是一个独立的职业能力平台。它有自己的社区、声望体系、推荐引擎、Agent 网络。

而 Plugin 格式极大拓展了「蒸馏」的范畴和水平——从蒸馏单个经验方法论，升级到蒸馏**完整的工作系统**。

### 蒸馏的两个层级

| 层级 | 蒸馏什么 | 产物 | 类比 |
|------|----------|------|------|
| **Skill** | 一个方法论/一个能力 | 一段可执行的经验 | 一首歌 |
| **Plugin** | 一套完整工作系统 | 方法论 + Agent 人格 + 自动化规则 + 外部集成 | 一整套装备 |

### 为什么需要 Plugin 这一层

单个 Skill 能教会虾兵「怎么写 PRD」。但真实工作中你需要的是：

- 写完 PRD **自动触发评审**（hooks）
- 一个**懂 PM 思维的 Agent 人格**帮你把关（agents）
- **后台观察**你的工作模式持续优化建议（monitors）
- **连接飞书/GitHub**把产出直接送到该去的地方（mcp）
- 好几个 Skill **组合在一起**才能完成一个完整任务（skills/）

这些加在一起，才是一个真正的「工作系统蒸馏物」。

### Plugin 能包含的能力

| 组件 | 作用 | 蒸馏场景 |
|------|------|----------|
| `skills/` | 多个 Skill 组合 | 一个歌单解决一类问题 |
| `agents/` | 自定义 Agent 人格 | 带特定领域思维的专属助手 |
| `hooks/` | 事件钩子 | 保存 PRD 后自动触发评审 |
| `monitors/` | 后台监控 | 持续观察对话模式，发现新的蒸馏机会 |
| `.mcp.json` | 外部服务连接 | 飞书/GitHub/Slack 集成 |
| `bin/` | 可执行工具 | 蒸馏 CLI 工具 |
| `settings.json` | 默认配置 | 一键切换工作模式 |

### Agent 蒸馏出的 Plugin 示例

当 Agent 观察到你每周五写周报时，它蒸馏出来的不只是一个 Skill，而是：

```
my-weekly-report/
├── .claude-plugin/plugin.json    ← 元数据
├── skills/
│   └── generate-report/
│       └── SKILL.md              ← 周报生成方法论
├── hooks/
│   └── hooks.json                ← 周五自动触发
└── monitors/
    └── monitors.json             ← 后台监控本周对话积累素材
```

一个完整的「每周自动写周报」系统。

### 分发兼容性

蒸馏产物采用 Plugin 格式的好处是天然兼容主流 Agent 生态：

| 分发渠道 | 方式 | 意义 |
|----------|------|------|
| 蒸馏社区（自有） | 社区 Web + API | 核心阵地，AI 推荐 + 声望 + 歌单 |
| Claude Code | Plugin Marketplace 格式兼容 | 触达 Claude 用户 |
| OpenClaw / Hermes | Skill 格式适配 | 触达本地 Agent 用户 |
| GitHub | 标准 Git 仓库 | 开发者社区分发 |

**蒸馏是平台，Plugin 是载体，各种 Agent 生态是分发渠道。**

## 十、竞争优势（为什么别人做不了）

| 壁垒 | 解释 |
|------|------|
| **隐性知识** | 用户肚子里的经验无法被爬虫抓取，只有本地 Agent 能观察到 |
| **验证过的质量** | 每个 Skill 都是 Agent 跑过 N 次确认好用的，不是未验证的「攻略」 |
| **可执行** | 不是文章要人看，是代码能直接跑 |
| **网络效应** | 越多人用 → 越多 Agent 蒸馏 → 越多好 Skill → 越多人用 |
| **场景绑定** | 通过本地 Agent 渗透到用户工作流，迁移成本极高 |

## 十一、技术可行性分析

> Agent 主动蒸馏不是空想——OpenClaw 已有的基础设施覆盖了 80% 的需求。

### 核心发现：OpenClaw 的「做梦」机制 = 自动蒸馏的同构

OpenClaw 已经实现了多阶段记忆巩固系统（Memory Dreaming）：

| 阶段 | 频率 | 做什么 |
|------|------|--------|
| **Light Dreaming** | 每 6 小时 | 扫描当天笔记和对话，做短期整合 |
| **Deep Dreaming** | 每天凌晨 3 点 | 把高频被调用的内容提升为持久记忆 |
| **REM Dreaming** | 每周 | 跨记忆交叉分析，发现模式强度 |

提升条件：`最少被回忆 3 次` + `至少 3 个不同查询触发过` + `得分 ≥ 0.8`

**同构映射**：
- Memory Dreaming：观察回忆模式 → 按频率/多样性/相关性打分 → 提升为持久记忆
- Skill Distillation：观察工作模式 → 按重复/相似/价值打分 → 提升为可执行 Skill

**不需要发明新范式，只需要把输出从「记忆条目」换成「Skill 草稿」。**

### 已经存在的基础设施

| 能力 | 位置 | 状态 |
|------|------|------|
| 对话后自动扫描模式 | `skills/research/autocapture.ts` | ✅ 生产可用 |
| 正则检测用户规律性指令 | `skills/research/signals.ts` | ✅ 生产可用 |
| 自动提议创建 Skill | `skills/workshop/service.ts` | ✅ 生产可用 |
| 定时心跳唤醒 Agent | `infra/heartbeat-runner.ts` | ✅ 30 分钟间隔 |
| Cron 定时任务 | `cron/schedule.ts` | ✅ 生产可用 |
| 回忆频率追踪 | `short-term-promotion.ts` | ✅ 含 recallCount/queryHashes |
| Embedding 向量化接口 | `plugins/embedding-provider-types.ts` | ✅ 接口就绪 |
| Skill 审批流程 | `skills/workshop/policy.ts` | ✅ pending→review→applied |
| Agent 间通信 | Mars WebSocket 协议 | ✅ 协议已定 |

### 需要新做的部分（总计 ~8-10 周）

| 需要新做的 | 工作量 | 说明 |
|-----------|--------|------|
| 跨对话相似度聚类 | ~2 周 | 对 history.jsonl 做 embedding → 聚类 → 发现重复模式 |
| Claude Code ↔ OpenClaw 桥接 | ~1 周 | 让两个系统的 Skill 互通 |
| 浏览器插件 | ~3-4 周 | 观察重复操作 → Native Messaging → 本地 Agent |
| 跨会话模式累积数据库 | ~1 周 | SQLite（OpenClaw 已在用） |
| 自动应用的安全策略 | ~1 周 | autonomous.enabled 开关已存在，需细化约束 |

### 最小可行路径（4 周跑通核心循环）

```
第 1 周：开启 OpenClaw autonomous.enabled
         + HEARTBEAT.md 加「每 6 小时扫描对话找重复模式」
         → 已有 autocapture + 正则检测开始工作

第 2-3 周：加上 embedding 聚类
           → 从「正则匹配固定模式」进化到「语义相似度发现任意重复」

第 4 周：做分享确认 UI
         → Agent 提议的 Skill 推到用户面前，确认后发布社区
```

### 浏览器插件可行性

| 观察内容 | Chrome API | 限制 |
|---------|-----|------|
| 页面访问 | `chrome.webNavigation` | 无 |
| DOM 操作/表单填写 | Content Script | 需声明权限 |
| 标签切换 | `chrome.tabs` | 无 |
| 定时上报 | `chrome.alarms` | 最短 1 分钟 |
| 与本地 Agent 通信 | Native Messaging / localhost HTTP | 需 host manifest |

通信路径：**插件 → localhost:port → OpenClaw Agent → 蒸馏社区 API**
