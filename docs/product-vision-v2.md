> 蒸馏世界，成就自己！蒸馏自己，成就世界！

**日期**：2026 年 6 月 25 日  
**版本**：v2（统一梳理版）

## 一句话定义

**蒸馏是一个 AI Native 的职业能力平台。** 用户的本地 Agent 观察工作 → 自动蒸馏经验为可执行的能力包 → 帮用户干活 → 分享到社区帮助更多人。

---

## 一、产品哲学

### 1. 人都是懒的

主动蒸馏自己的人永远是少数。所以不要指望用户来找我们，而是 Agent 主动去发现、主动去蒸馏、主动去帮忙。

### 2. 蒸馏物是活的

蒸馏出来的东西不是文章，不是用来看的。它能被 Agent 直接执行，用户可能完全无感就享受到了好处。效果可度量——帮你省了多少时间、执行了多少次、成功率多少。

### 3. 分享是零成本的

用户不需要「创作」。Agent 已经帮你蒸馏好了、跑验证了、写好描述了。你只需要在 Agent 问你的时候点一下「好，分享」。

### 4. 智能发生在边缘

所有推理、匹配、打分、观察都发生在用户自己的 Agent 里，不在云端平台。平台是仓库和交易市场，真正的智能跑在离用户最近的地方。

这意味着：
- **隐私天然有保障** — 工作数据不需要上传，Agent 在本地完成分析
- **个性化无上限** — 每个人的 Agent 基于自己的上下文做决策
- **离线也能工作** — 蒸馏物装到本地后，不依赖平台也能执行
- **平台不是瓶颈** — 10 万用户的 Agent 各自在本地跑，平台不需要承担推理算力

## 二、核心概念：蒸馏物的两个层级

蒸馏产出的东西，我们统一称为**蒸馏物**。它有两个层级：

| 层级 | 是什么 | 蒸馏什么 | 类比 |
|------|--------|----------|------|
| **Skill** | 单个可执行的经验方法论 | 一个能力点 | 一首歌 |
| **Plugin** | 完整的工作系统 | Skill + Agent 人格 + 自动化规则 + 外部集成 | 一整套装备 |

### Skill：经验的最小蒸馏单元

一个 Skill 回答的是「**怎么做某件事**」——比如怎么写 PRD、怎么做竞品分析、怎么设计 A/B 实验。它是一段结构化的、可被 Agent 执行的经验。

### Plugin：工作系统的完整蒸馏

真实工作中，一个能力点往往不够。你需要的是：

- 写完 PRD **自动触发评审**（事件钩子）
- 一个**懂 PM 思维的 Agent 人格**帮你把关（专属 Agent）
- **后台观察**你的工作模式持续优化建议（后台监控）
- **连接飞书/GitHub** 把产出直接送到该去的地方（外部集成）
- 好几个 Skill **组合在一起**才能完成一个完整任务（能力组合）

这些加在一起就是一个 Plugin——一个「工作系统蒸馏物」。

### Plugin 的构成

| 组件 | 作用 | 举例 |
|------|------|------|
| `skills/` | 多个 Skill 组合 | 一个工具包解决一类完整问题 |
| `agents/` | 自定义 Agent 人格 | 带 PM 思维的专属助手 |
| `hooks/` | 事件钩子 | 保存文件后自动触发下一步 |
| `monitors/` | 后台监控 | 持续观察对话模式，发现新的蒸馏机会 |
| `mcp` | 外部服务连接 | 飞书/GitHub/Slack 集成 |
| `bin/` | 可执行工具 | 命令行工具 |
| `settings` | 默认配置 | 一键切换工作模式 |

## 四、四条核心链路

### 链路 1：Agent 主动蒸馏（供给侧）

```
Agent 持续观察用户工作（对话历史 / 浏览行为 / 工作节奏）
    ↓
发现重复模式 → 自主蒸馏成 Skill 或 Plugin 草稿
    ↓
先自己拿来帮用户干活（悄悄验证好不好用）
    ↓
跑了 N 次、用户没改过 → 确认好用
    ↓
Agent：「这个已经帮你用了 8 次了，社区有人需要，分享出去？」
    ↓
用户确认 → 发布到平台 → 声望 ↑
```

**关键设计**：不是蒸馏完就问「要不要公开」，而是**验证好用之后**再问。

### 链路 2：Agent 主动找蒸馏物（消费侧）

```
Agent 在帮用户干活
    ↓
意识到「这个事有更好的方法」或「我没做过」
    ↓
实时查询蒸馏平台 API
    ↓
平台返回匹配的蒸馏物（或多个的组合方案）
    ↓
Agent 直接装上就用 / 或组合执行
    ↓
用户甚至可以无感——活干完了就行
```

### 链路 3：定时任务 → 场景预判 → 直推本地 Agent

```
Agent 的定时任务触发（如：周五 14:00）
    ↓
Agent 知道「用户每周五写周报」
    ↓
去平台拉取「周报生成」Skill + 基于本周对话预生成草稿
    ↓
直接在对话里说：「周报写好了初稿，你看看改改？」
```

**不是推通知，不是发邮件，是活已经干完了直接送到你面前。**

推送通过本地 Agent（OpenClaw/Hermes）的心跳机制直达用户——不经过网页、不经过 App。

### 链路 4：AI 搜索 + 组合编排（问题驱动）

```
用户对 Agent 说：「我下周要做新产品立项评审」
    ↓
Agent 理解意图 → 查询平台 AI 推荐 API
    ↓
平台返回组合方案：
  1.「先写新闻稿」— 理清价值主张
  2.「市场规模测算」— 算清楚盘子
  3.「做一份惊艳的演示」— 包装成 Deck
    ↓
Agent：「我帮你按这个顺序走一遍？」
    ↓
用户：「好」→ Agent 依次执行，产出完整立项材料
```

## 五、分享飞轮

### 为什么用户会愿意分享？

| 动力 | 设计 |
|------|------|
| **零成本** | Agent 已经帮你蒸馏好了，你只需点「确认」 |
| **已验证** | 不是「可能有用」，是「确实帮我用了 N 次」 |
| **有人需要** | Agent 会告诉你「社区有 X 人在找类似的」 |
| **声望正反馈** | 分享后被引用 → 排名升 → 更有动力 |
| **反哺自己** | 别人 Fork 改进了 → 你也能受益 |

### 分享确认时机

Agent 在以下条件**同时满足**时询问：

1. ✅ 蒸馏物已被自动执行 ≥ 3 次
2. ✅ 用户未对执行结果做过修改（质量稳定）
3. ✅ 平台中存在相关需求（有人搜索过类似问题）

> 「这个'客户投诉处理三步法'我已经帮你跑了 5 次了，每次你都没改直接用了。社区里有 12 个人在找类似的，分享出去帮帮他们？你的声望会 +50。」

## 六、内容组织：赛道 + 工具包

### 职业赛道（官方视角）

按职业组织一切：产品经理 / 工程师 / 设计师 / 运营 / HR / ...

每个赛道有子能力域，用户进来就知道「我该去哪变强」。一个蒸馏物可以同时属于多个赛道。

### 工具包（用户视角）

像网易云音乐工具包一样，蒸馏物可以被自由组合为收藏集。

| 工具包类型 | 示例 |
|----------|------|
| **新手入门包** | 「新手产品经理必备 10 个 Skill」 |
| **场景组合包** | 「从 0 到 1 做一次完整的产品立项」 |
| **大佬配置单** | 「Top 1 PM 的虾兵装备清单」 |
| **公司/团队包** | 「XX 团队新人 Onboarding 必装」 |
| **AI 智能推荐** | Agent 基于你的能力缺口自动生成的个性化工具包 |

工具包互动：一键安装 / Fork / 订阅更新 / 引用计入声望。

Agent 还可以：
- 基于能力缺口**自动生成个性化工具包**
- 发现你装了工具包中 3 个，**提醒装剩下的**
- 帮你**自动创建工具包**：「你最常用这 8 个，打包分享？」

## 七、信息采集：Agent 如何观察用户

| 渠道 | 方式 | 能发现什么 |
|------|------|-----------|
| **对话历史** | 本地 Agent memory | 重复的提问模式、相似的任务 |
| **浏览器插件** | 页面行为追踪 | 每天重复操作的工作流 |
| **日历/飞书** | API 集成 | 周期性事件（周报、月会、复盘） |
| **文件系统** | 本地扫描 | 反复创建的相似文档模板 |
| **IDE 插件** | 代码行为 | 重复的编码模式、Review 习惯 |

**隐私原则**：所有观察数据留在本地，只有用户确认分享的蒸馏物才上传平台。

## 八、分发兼容性

蒸馏是独立平台。Plugin 格式是技术载体。各种 Agent 生态是分发渠道。

| 分发渠道 | 方式 | 意义 |
|----------|------|------|
| **蒸馏平台（自有）** | 社区 Web + API | 核心阵地：AI 推荐 + 声望 + 工具包 |
| **Claude Code** | 兼容 Plugin Marketplace 格式 | 触达 Claude 用户 |
| **OpenClaw / Hermes** | Skill 格式适配 | 触达本地 Agent 用户 |
| **GitHub** | 标准 Git 仓库 | 开发者社区分发 |

蒸馏产物采用 Plugin 格式的好处：天然兼容主流 Agent 生态，不需要每个渠道单独适配。但蒸馏平台本身的价值在于社区、推荐、声望、网络效应——不在于某一种格式或某一个渠道。

## 十一、竞争优势

| 壁垒 | 解释 |
|------|------|
| **隐性知识** | 用户肚子里的经验无法被爬虫抓取，只有本地 Agent 能观察到 |
| **验证过的质量** | 每个蒸馏物都是 Agent 跑过 N 次确认好用的 |
| **可执行** | 不是文章要人看，是能直接跑的能力 |
| **网络效应** | 越多人用 → 越多 Agent 蒸馏 → 越多好内容 → 越多人用 |
| **场景绑定** | 通过本地 Agent 渗透到工作流，迁移成本极高 |
| **边缘智能** | 不依赖平台算力，天然可扩展 |

## 十二、技术可行性

> Agent 主动蒸馏不是空想——OpenClaw 已有的基础设施覆盖了 80% 的需求。

### 核心发现

OpenClaw 已实现多阶段「Memory Dreaming」系统（每 6 小时/每天/每周巩固记忆）。这套机制跟自动蒸馏在概念上完全同构——只需要把输出从「记忆条目」换成「蒸馏物草稿」。

### 已有基础设施

| 能力 | 状态 |
|------|------|
| 对话后自动扫描模式 | ✅ 生产可用 |
| 正则检测用户规律性指令 | ✅ 生产可用 |
| 自动提议创建 Skill | ✅ 生产可用 |
| 定时心跳唤醒 Agent（30分钟） | ✅ 生产可用 |
| Cron 定时任务 | ✅ 生产可用 |
| 回忆频率追踪 | ✅ 生产可用 |
| Embedding 向量化接口 | ✅ 接口就绪 |
| Skill 审批流程 | ✅ pending→review→applied |
| Agent 间通信（WebSocket） | ✅ 协议已定 |

### 需要新做的（~8-10 周）

| 工作 | 工作量 |
|------|--------|
| 跨对话相似度聚类 | ~2 周 |
| Claude Code ↔ OpenClaw 桥接 | ~1 周 |
| 浏览器插件 | ~3-4 周 |
| 跨会话模式累积数据库 | ~1 周 |
| 自动应用安全策略 | ~1 周 |

### 最小验证路径（4 周）

1. **第 1 周**：开启 OpenClaw autonomous.enabled + 心跳扫描 → 正则检测开始工作
2. **第 2-3 周**：加 embedding 聚类 → 语义相似度发现任意重复模式
3. **第 4 周**：分享确认 UI → 完成蒸馏到发布的闭环
