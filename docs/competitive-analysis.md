## 〇、分析框架说明

本报告不再按「竞品逐个介绍」的传统框架，而是从用户的底层需求出发，画出蒸馏的竞争全景：

```
用户的底层需求：「让我少干重复的活，把经验变成自动化」

解决这个需求的所有路径：
                                                    
 ① 自己动手配置        → Cursor Rules / CLAUDE.md / Shortcuts
 ② 买别人做好的        → GPT Store / Coze Store / App Store
 ③ 雇 AI 帮我从头做    → Replit Agent / Claude Code / v0
 ④ AI 自动帮我积累     → 蒸馏 Distill（新品类）
```

蒸馏是路径 ④ 的开创者——**AI 主动从行为中蒸馏**。但它不是在真空中竞争，每一条路径都在争夺用户的同一份注意力。

## 一、需求拆解：用户到底要什么

### 1.1 用户需求四层模型

| 层级 | 用户需求 | 翻译成产品语言 | 现有满足者 |
|------|---------|-------------|-----------|
| **L1: 执行** | 「帮我把这活干了」 | Agent 直接执行任务 | Claude Code, Replit, Copilot |
| **L2: 复用** | 「下次别让我再说一遍」 | 经验沉淀 + 自动触发 | Cursor Rules, CLAUDE.md, Shortcuts |
| **L3: 发现** | 「我不知道有更好的方法」 | 智能推荐 + 能力补全 | GPT Store, Coze, Copilot Extensions |
| **L4: 进化** | 「自动帮我变强」 | 持续学习 + 能力网络 | **无（蒸馏的机会）** |

> **核心洞察**：当前所有竞品都卡在 L1-L3。没有产品在做 L4——「Agent 主动帮用户进化」。蒸馏的战略定位就是占领 L4，同时向下兼容 L1-L3。

### 1.2 四条路径的本质对比

| 路径 | 创造者 | 触发方式 | 进化性 | 用户成本 |
|------|--------|---------|--------|---------|
| ① 自己配置 | 用户 | 手动 | 静态 | 高（需技术能力） |
| ② 买现成的 | 第三方 | 手动搜索 | 静态 | 中（需选择判断） |
| ③ AI 从头做 | AI（指令驱动） | 用户发起 | 无 | 低（但每次从头来） |
| **④ AI 自动积累** | **AI（行为驱动）** | **自动** | **持续进化** | **零** |

## 三、第一圈：直接品类竞争

### 3.1 OpenAI GPT Store

**本质**：AI 能力的「应用商店」——人创建，人搜索，对话式使用。

| 维度 | 分析 |
|------|------|
| **产品形态** | 用户用对话创建 GPT → 发布到 Store → 其他人搜索使用 |
| **最新动态** | 2026 年支持 GPT Actions 调用外部 API；GPT-5 增强推理；收入分成正式上线 |
| **用户规模** | 1.5 亿+ ChatGPT 用户，数百万 GPT 被创建 |
| **核心优势** | 海量用户基数 · 创建门槛极低 · OpenAI 品牌 · 生态闭环 |
| **致命弱点** | 不可执行（只是对话）· 质量洪水 · 静态（不学习不进化）· 封闭生态 |

**与蒸馏的根本差异**：

| | GPT Store | 蒸馏 |
|---|-----------|------|
| 供给侧 | 人手动创建 | Agent 自动蒸馏 |
| 质量保障 | 无（发了就上）| 验证后才发布 |
| 使用方式 | 对话（你说它做）| 执行（Agent 直接干完）|
| 进化性 | 静态（一次定稿）| 活的（持续迭代）|
| 触达用户 | 用户搜索（被动）| Agent 推送（主动）|

**竞争判断**：GPT Store 是蒸馏最容易讲清差异的对标物——「GPT Store 是手动创建 + 对话使用，蒸馏是自动蒸馏 + 直接执行」。用户教育时可借其心智。

---

### 3.2 字节跳动 Coze（扣子）

**本质**：AI Bot 开发平台——提供可视化工作流 + 插件，让用户搭建复杂 Bot。

| 维度 | 分析 |
|------|------|
| **产品形态** | 低代码 Bot Builder + 工作流编排 + 多渠道发布（飞书/微信/Web/API）|
| **最新动态** | 2026 年接入豆包大模型；工作流支持 Agent 间调用；企业版大规模推广 |
| **用户规模** | 数百万创建者，字节体系内大量企业客户 |
| **核心优势** | 工作流可视化 · 多渠道分发 · 字节流量加持 · 企业级能力 |
| **致命弱点** | 仍是「主动创建」模式 · 以 Bot 为中心而非以人为中心 · 云端依赖 · 学习成本不低 |

**与蒸馏的根本差异**：

| | Coze | 蒸馏 |
|---|------|------|
| 中心 | Bot（工具导向）| 人（能力导向）|
| 创建方式 | 用户搭建工作流 | Agent 自动从行为蒸馏 |
| 使用场景 | 搭好后被动等人用 | 嵌入日常，主动干活 |
| 知识来源 | 外部数据（API/文档）| 隐性知识（本地行为）|
| 运行位置 | 云端 | 本地优先 |

**竞争判断**：Coze 有执行力但缺乏「自动发现」能力。它是给「愿意搭建」的人用的；蒸馏是给「不想费事但想变强」的人用的。用户群有差异，但 Coze 如果加入行为观察能力，会变成最强威胁。

---

### 3.3 Google Gemini Gems

**本质**：定制化 Gemini 实例——上传文档 + 设定人设 = 专属对话 AI。

| 维度 | 分析 |
|------|------|
| **产品形态** | 自定义指令 + 知识文件上传 + 链接分享 |
| **最新动态** | 2026 年 Gemini 2.0 增强多模态；与 Google Workspace 深度集成 |
| **用户规模** | Google One AI Premium 用户（千万级）|
| **核心优势** | Google 生态（Drive/Gmail/Docs/Calendar）深度集成 |
| **致命弱点** | 本质只是自定义对话 · 无执行能力 · 无社区机制 · 生态封闭 |

**竞争判断**：Gems 对蒸馏的直接威胁较低——产品成熟度和能力深度不足。但 Google 有海量用户行为数据（Gmail/Calendar/Drive），如果将来做行为分析 + 自动化，有数据优势。

## 四、第二圈：跨品类竞争

### 4.1 Claude Code + MCP 生态

**本质**：开发者本地 Agent——真正能执行代码/命令/文件操作的 AI。

| 维度 | 分析 |
|------|------|
| **最新动态** | 2026 年 MCP 成为事实标准；Plugin 格式稳定；支持自定义 Agent 配置 |
| **核心优势** | 真正可执行 · 本地 Agent（文件+终端）· MCP 开放协议 · 开发者信任度高 |
| **致命弱点** | 纯开发者工具 · 无自动蒸馏 · 无社区发现机制 · 需手动配置 |

**与蒸馏的关系**：**互补 > 竞争**

蒸馏的 Plugin 格式兼容 Claude Code 生态——Claude Code 是蒸馏最重要的分发渠道之一。蒸馏补全 Claude Code 缺失的社区层（推荐 + 声望 + 自动发现）。但如果 Anthropic 自己做社区市场 + 自动蒸馏能力，这个关系会从互补变为竞争。

---

### 4.2 AI IDE 群体（Cursor / Windsurf / Zed AI）

**本质**：编程场景的 AI 增强——内嵌于编辑器的代码生成和理解能力。

| 维度 | 分析 |
|------|------|
| **Cursor 最新** | Rules 生态活跃；Agent 模式可自主执行；Context 系统理解全局代码 |
| **Windsurf 最新** | Cascade 多步执行；Flows 工作流；Memories 记忆系统（⚠️ 最接近蒸馏的概念）|
| **核心优势** | 深度嵌入开发流程 · 效果可感知 · 社区活跃（Cursor Rules） |
| **致命弱点** | 仅覆盖开发者 · 规则静态 · 不跨场景 · 无法覆盖非编码工作 |

**⚠️ 特别关注 Windsurf Memories**：

Windsurf 的 Memories 功能会从对话中自动学习用户偏好——这与蒸馏的「Agent 观察 → 自动学习」理念高度相似。区别在于：
- Memories 只学「偏好」（你喜欢 Tab Size 2），不学「方法论」（你怎么做竞品分析）
- Memories 只服务自己，不能分享到社区
- Memories 只在编程场景，不覆盖全职业

但它验证了一个信号：**市场认可「AI 主动学习用户」这个方向**。

---

### 4.3 Apple Intelligence / Shortcuts

**本质**：设备端 AI + 自动化——Apple 生态内的智能助手和工作流自动化。

| 维度 | 分析 |
|------|------|
| **最新动态** | 2026 年 Apple Intelligence 大幅增强；App Intents 让第三方 App 可被 Siri 调度 |
| **核心优势** | 设备端运行（隐私）· 系统级权限（全 App 数据）· 10 亿+ 设备 |
| **致命弱点** | AI Agent 能力极弱 · 无社区分享生态 · 自动化能力浅 · 封闭系统 |

**竞争判断**：Apple 有最强的本地数据访问权限（短信/日历/邮件/位置/应用使用），但其 AI 能力和开放性严重不足。如果 Apple 在 Shortcuts 中加入「自动发现重复模式 → 建议自动化」，将是蒸馏在消费端的强力对手。短期风险低，长期需观察。

## 五、第三圈：生态位竞争

### 5.1 Microsoft Copilot + Graph

| 维度 | 威胁分析 |
|------|---------|
| **数据优势** | Microsoft Graph 掌握企业全量行为数据（邮件/会议/文档/Teams 消息）|

| **缺失点** | 企业级定位意味着无法做个人化的「隐性知识蒸馏」；合规约束重 |
| **威胁等级** | ⚠️ 中等（在企业场景有数据优势，但基因决定不会做 C 端社区）|

| 维度 | 威胁分析 |

| **威胁等级** | ⚠️ 中等（有数据无隐私，有执行无社区）|

| 维度 | 威胁分析 |
|------|---------|
| **优势** | 在各自场景内有最深的用户行为数据 |
| **缺失点** | 各自为政，不能跨场景 · 做的是场景内辅助，不是经验积累 |
| **威胁等级** | 🟡 低（各自场景做得很好但不跨界，不构成平台级竞争）|

## 七、关键战场分析

### 7.1 战场一：供给侧之战

**问题**：谁能以更低成本产生更高质量的 AI 能力？

| 模式 | 供给成本 | 质量天花板 | 规模化潜力 |
|------|---------|-----------|-----------|
| 用户手动创建（GPT Store/Coze）| 高 | 低（大量低质）| 中（有人口上限）|
| 开发者贡献（Plugin 市场）| 中 | 高 | 低（开发者有限）|
| **Agent 自动蒸馏（蒸馏）** | **零** | **高（验证过）** | **高（随用户规模线性增长）**|

> **结论**：蒸馏的供给侧逻辑是碾压性的——每多一个活跃用户，就多一台自动蒸馏机器。GPT Store 要 100 个人来创建 100 个 GPT；蒸馏只需要 100 个人正常工作，Agent 自己产出 100 个 Skill。

### 7.2 战场二：执行力之战

**问题**：产出的能力包能不能真正「干活」？

| 产品 | 执行力类型 | 场景范围 | 执行可靠性 |
|------|-----------|---------|-----------|
| GPT Store | ❌ 对话（不能执行）| 全场景（但都是对话）| N/A |
| Coze | ⚠️ 工作流（有限执行）| 线上场景 | 中 |
| Claude Code | ✅ 本地 Agent（全能执行）| 开发场景 | 高 |
| **蒸馏** | ✅ 本地 Agent（全能执行）| 全职业场景 | 高（经过验证）|

### 7.3 战场三：发现效率之战

**问题**：用户怎么找到适合自己的能力？

| 模式 | 发现效率 | 精准度 | 用户体验 |
|------|---------|--------|---------|
| 搜索浏览（App Store 模式）| 低 | 低 | 累（要自己翻）|
| 推荐算法（抖音模式）| 中 | 中 | 被动接收 |
| **Agent 场景匹配（蒸馏模式）** | **高** | **高** | **无感（活干完了）**|

> **蒸馏的发现机制是革命性的**：用户甚至不需要知道蒸馏物的存在——Agent 在工作中实时匹配、静默安装、直接使用。

### 7.4 战场四：网络效应之战

**问题**：谁能建立最强的增长飞轮？

```
GPT Store 飞轮（弱）：
用户多 → 创建者多 → GPT 多（但质量差）→ 搜索体验差 → 用户不满

Coze 飞轮（中）：
字节流量 → 创建者来 → Bot 多 → 多渠道分发 → 更多用户

蒸馏飞轮（强，但需时间）：
用户多 → Agent 蒸馏多 → 高质量 Skill 多 → 其他用户的 Agent 调用
→ 他们也蒸馏 → 正向循环 → 每个用户既是消费者也是（无感的）生产者
```

**飞轮质量对比**：
- GPT Store：供给质量不可控 → 飞轮越转质量越差
- Coze：依赖字节流量输血 → 不是有机飞轮
- **蒸馏：每个用户自动贡献 → 规模越大质量越高 → 最强飞轮结构**

## 八、用户旅程对比

### 一个产品经理做竞品分析的完整旅程

| 阶段 | GPT Store | Coze | 蒸馏 |
|------|-----------|------|------|
| **触发** | 用户想到去搜索 | 用户想到去搜索 | Agent 看到你打开了竞品网站，主动问"要做竞品分析？" |
| **选择** | 翻 100 个 GPT 不知选哪个 | 在 Bot 列表里找 | Agent 直接说"上次你用这套方法效果不错" |
| **使用** | 对话来回说需求 | 填表 → 跑工作流 | Agent 直接执行，5 分钟后给你初稿 |
| **产出** | 一段对话文字 | 结构化文档 | 已格式化的报告 + 插入到你正在用的文档 |
| **沉淀** | 无 | 无 | Agent 记住了这次的改动，下次更好 |
| **分享** | 你得手动创建一个 GPT | 你得搭一个 Bot | Agent: "这方法你改进了 3 次都没再动过，分享？" 你: "好" |

| Coze/飞书 加入行为观察蒸馏 | 中 | 12-18月 | 🔴 极高 | 字节有数据但有隐私公关风险，本地 Agent 做不了 |
| Windsurf 将 Memories 扩展到全场景 | 中 | 6-12月 | ⚠️ 高 | 蒸馏做社区分享，Windsurf 只做本地个人化 |
| Apple 用设备端 AI 做自动化建议 | 低 | 18-24月 | 🟡 中 | Apple 不做社区，且 Agent 能力薄弱 |
| 某创业公司做同品类竞争 | 中 | 随时 | 🟡 中 | 先发优势 + 网络效应 + 技术壁垒 |

### 9.2 最大威胁深度分析

**Scenario A：Anthropic 自建社区市场**

如果 Anthropic 在 Claude Code 上推出：
- Plugin 市场 + 推荐引擎 + 用户评分系统

蒸馏的应对：
- Claude Code 只覆盖开发者 → 蒸馏覆盖全职业
- 没有「自动蒸馏」→ 仍是用户手动分享
- 蒸馏可以作为 Claude Code Plugin 生态的超集
- **核心策略：做 Claude Code 的「能力商店」，而非竞争对手**

**Scenario B：字节用飞书数据做行为蒸馏**

如果字节做出：
- 从飞书聊天/文档/审批中自动提取工作模式 → 生成自动化

蒸馏的应对：
- 飞书方案 = 云端数据分析，隐私风险极高 → 用户抵触
- 绑定飞书生态，不跨平台 → 换工具就没了
- 蒸馏 = 本地 Agent，数据不上传 → 隐私优势
- **核心策略：打「隐私牌」——你的经验在你的设备上，不在别人的服务器上**

## 十三、最终结论

### 核心判断

1. **品类优势明确**：蒸馏不是在红海中抢份额，而是定义了「AI 自动蒸馏经验」这个新品类。在核心能力维度上，没有竞品覆盖。

2. **最大风险不是竞品，是自己**：
   - 能否在 6 个月内证明「自动蒸馏」可行且好用
   - 能否在种子用户中跑通网络效应
   - 能否在大厂注意到之前建立足够深的壁垒

3. **最大机会在互补**：与 Claude Code / OpenClaw / Hermes 做互补而非竞争，成为它们的「能力后端」，借力增长。

4. **时间窗口 12-18 个月**：大厂注意到这个方向并能出产品大概需要 12-18 个月（内部立项 → 开发 → 发布）。蒸馏需要在此之前建立不可逆的网络效应。
