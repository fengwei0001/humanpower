一

核心定位

蒸馏不是一个网站，是一个 Agent 网络的能力后端。

传统社区让用户来逛。蒸馏让 Agent 替用户干活。

网站只是仓库和交易市场，真正的产品体验发生在用户自己的 Agent 里。

二

产品哲学

😴

人都是懒的

主动蒸馏自己的人永远是少数。Agent 观察用户 → 主动发现模式 → 自己蒸馏 → 拿给用户确认。

⚡

Skill 是活的

不是文章，不是用来看的。能被 Agent 直接执行，用户可能无感就享受到了好处。效果可度量。

🎁

分享是零成本的

Agent 帮你蒸馏好了、跑验证了、写好描述了。你只需要点一下「好，分享」。

🧠

智能发生在边缘

所有推理、匹配、打分、观察都发生在用户自己的 Agent 里。平台是仓库和交易市场，真正的智能跑在离用户最近的地方。

四

四条核心链路

🔄 链路 1：Agent 主动蒸馏（供给侧）
  • Agent 持续观察用户工作（对话历史 / 浏览行为 / 工作节奏）
  • 发现重复模式 → 自主蒸馏成 Skill 草稿
  • 先自己拿来帮用户干活（悄悄验证好不好用）
  • 跑了 N 次、用户没改过 → 确认好用
  • Agent 问：「这个已经帮你用了 8 次了，社区有人需要，分享出去？」
  • 用户确认 → 发布到社区 → 声望 ↑
⭐ 关键设计：不是蒸馏完就问「要不要公开」，而是验证好用之后再问。
🔍 链路 2：Agent 主动找 Skill（消费侧）
  • Agent 在帮用户干活时，意识到「这个事有更好的方法」
  • 实时 ping 蒸馏社区 API
  • 社区返回匹配的 Skill（或多个 Skill 的组合方案）
  • Agent 直接装上就用 — 用户甚至可以无感
⏰ 链路 3：定时任务 → 场景预判
  • Agent 的定时任务触发（如：周五 14:00）
  • Agent 知道「用户每周五写周报」
  • 去社区拉取「周报生成」Skill + 基于本周对话历史预生成草稿
  • 直接推送：「周报写好了初稿，你看看改改？」
⭐ 用户收到的不是「推荐你一个 Skill」，而是活已经干完了。推送通过本地 Agent（OpenClaw/Hermes）的心跳机制直达用户。
🧩 链路 4：AI 搜索 + Skill 组合（问题驱动）
  • 用户对 Agent 说：「我下周要做新产品立项评审」
  • Agent 调用社区 AI 推荐 API → 返回组合方案
  • ① 「先写新闻稿」理清价值 → ② 「市场规模测算」算清盘子 → ③ 「惊艳演示」包装成 Deck
  • Agent：「我帮你按这个顺序走一遍？」→ 用户：「好」→ 自动依次执行

分享机制设计

为什么用户会愿意分享？

动力设计
零成本Agent 已经帮你蒸馏好了，你只需点「确认」
已验证不是「可能有用」，是「确实帮我用了 N 次」
有人需要Agent 会告诉你「社区有 X 人在找类似的」
声望正反馈分享后被引用 → 排名升 → 更有动力
反哺自己别人 Fork 你的 Skill 改进了 → 你也能受益

分享确认时机

Agent 在以下条件同时满足时询问：

✅ Skill 已被自动执行 ≥ 3 次

✅ 用户未对执行结果做过修改（说明质量稳定）

✅ 社区中存在相关需求（有人搜索过类似问题）

「这个'客户投诉处理三步法'我已经帮你跑了 5 次了，每次你都没改直接用了。社区里有 12 个人在找类似的 Skill，分享出去帮帮他们？你的声望会 +50。」

九·五

从 Skill 到 Plugin：蒸馏的范畴升级

蒸馏是一个独立的职业能力平台。Plugin 格式极大拓展了「蒸馏」的范畴和水平——从蒸馏单个方法论，升级到蒸馏完整的工作系统。

🎯 定位：蒸馏是平台，Plugin 是载体，各种 Agent 生态是分发渠道。

蒸馏的两个层级

层级蒸馏什么产物类比
Skill一个方法论/能力一段可执行的经验一首歌
Plugin一套完整工作系统方法论 + Agent 人格 + 自动化规则 + 外部集成一整套装备

为什么需要 Plugin 这一层

单个 Skill 能教会虾兵「怎么写 PRD」。但真实工作中你需要的是：

📝 写完 PRD 自动触发评审（hooks）

🧠 一个懂 PM 思维的 Agent 人格帮你把关（agents）

👁️ 后台观察你的工作模式持续优化建议（monitors）

🔌 连接飞书/GitHub把产出直接送到该去的地方（mcp）

🧩 好几个 Skill 组合在一起才能完成一个完整任务（skills/）

这些加在一起，才是一个真正的「工作系统蒸馏物」。

Plugin 能包含的能力

组件作用蒸馏场景
skills/多个 Skill 组合一个歌单解决一类问题
agents/自定义 Agent 人格带特定领域思维的专属助手
hooks/事件钩子保存 PRD 后自动触发评审
monitors/后台监控持续观察对话模式，发现新的蒸馏机会
.mcp.json外部服务连接飞书/GitHub/Slack 集成
bin/可执行工具蒸馏 CLI 工具
settings.json默认配置一键切换工作模式

Agent 蒸馏出的 Plugin 示例

当 Agent 观察到你每周五写周报时，它蒸馏出来的不只是一个 Skill，而是一个完整的「每周自动写周报」系统：

my-weekly-report/ ├── .claude-plugin/plugin.json ← 元数据 ├── skills/ │ └── generate-report/ │ └── SKILL.md ← 周报生成方法论 ├── hooks/ │ └── hooks.json ← 周五自动触发 └── monitors/ └── monitors.json ← 后台监控本周对话积累素材

十一

技术可行性分析

Agent 主动蒸馏不是空想——OpenClaw 已有的基础设施覆盖了 80% 的需求。

🎯 核心发现：OpenClaw 已经有了「Agent 自动做梦巩固记忆」的完整管线。我们要做的只是让它做梦时除了巩固记忆，还学会蒸馏 Skill。

关键发现：OpenClaw 的「做梦」机制

OpenClaw 实现了多阶段记忆巩固系统（Memory Dreaming），跟我们的自动蒸馏在概念上完全同构：

阶段频率做什么
Light Dreaming每 6 小时扫描当天笔记和对话，做短期整合
Deep Dreaming每天凌晨 3 点把高频被调用的内容提升为持久记忆
REM Dreaming每周跨记忆交叉分析，发现模式强度

提升条件：最少被回忆 3 次 + 至少 3 个不同查询触发过 + 得分 ≥ 0.8

⭐ 同构映射：
Memory Dreaming：观察回忆模式 → 按频率/多样性/相关性打分 → 提升为持久记忆
Skill Distillation：观察工作模式 → 按重复/相似/价值打分 → 提升为可执行 Skill
